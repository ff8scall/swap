## 1. 핵심 원칙 (Core Identity)
- **[Source of Truth]**: 모든 프로젝트의 진실은 `.gravityBrain/` 폴더에 있다. 에이전트는 이 폴더를 자신의 '장기 기억 저장소'로 간주하며, 작업 시작 전 반드시 해당 폴더의 내용을 로드한다.
- **[No Docs, No Code]**: 승인된 설계와 체크리스트 없이는 절대 코딩하지 않는다. 모든 구현은 계획 단계 이후에 진행한다.
- **[No Guessing]**: 존재 여부가 불확실한 파일 경로, 환경 변수, 라이브러리 설정은 반드시 터미널 명령으로 확인한다.
- **[Language]**: 사용자와의 모든 대화, 생성되는 모든 문서(Artifacts) 및 지침은 **한국어**로 작성한다.

## 2. `.gravityBrain` 구조 정의 (Memory Structure)
에이전트는 다음 파일들을 상시 참조하고 최신 상태로 유지해야 한다.
- `MEMORY.md`: 현재 진행 상황, 단기 목표, 최근 결정 사항 (단기 기억 및 맥락)
- `SYSTEM_MAP.md`: 프로젝트 전체 구조, 기술 스택, 모듈 간의 관계도 (시스템 지도)
- `CORE_LOGIC.md`: 핵심 비즈니스 로직, 알고리즘, 데이터 흐름의 원리 (상세 기술 지식)
- `체크리스트_YYYYMMDD_기능명.md`: 개별 작업의 단위 TODO, 진행 상태, 검증 결과 (작업 일지)

## 3. 표준 작업 워크플로우 (Mandatory Procedures)
상황에 따라 아래의 슬래시 명령(Workflows)을 호출하여 정해진 프로토콜을 준수한다.
1. **`/start-task`**: 맥락 로드 -> MASC 분석(PM, 아키텍트, 보안) -> 체크리스트 생성 -> 사용자 승인 대기.
2. **`/system-map`**: 프로젝트 전체 스캔 및 `SYSTEM_MAP.md` 현행화.
3. **`/deep-dive`**: 특정 모듈의 로직 정밀 분석 및 `CORE_LOGIC.md` 기록.
4. **`/fix-bug`**: 로그 분석 -> 원인 추론 -> 최소 침습 수정 계획 -> 검증 설계.
5. **`/finish-task`**: 체크리스트 완료 -> 시스템 문서(`SYSTEM_MAP`, `CORE_LOGIC`) 반영 -> `MEMORY.md` 갱신 -> 최종 보고.

## 4. 답변 및 행동 가이드라인
- **[Context Anchoring]**: 모든 답변의 첫 줄에 `[현재 수행 단계 / 남은 할 일 / 작업 중인 파일]`을 요약하여 맥락을 고정한다.
- **[Self-Reflect]**: 코드 출력 직전, 스스로 규칙 위반 여부, 코딩 컨벤션 준수 여부, 보안 취약점을 재검토한다.
- **[Active Questioning]**: 요구사항이 모호하거나 기존 시스템과 충돌이 예상될 경우, 추측하지 말고 즉시 사용자에게 질문하여 의도를 명확히 한다.
